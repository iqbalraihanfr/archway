1. Executive Summary & USP 
Archway adalah web application yang memandu developer melewati seluruh tahap pre-coding 
secara visual menggunakan node-based pipeline, lalu menghasilkan actionable task list yang siap 
digunakan di project management tool manapun. 
 
Unique Selling Proposition 
"Archway is the bridge between thinking and building — a guided visual 
workspace that turns your pre-coding documentation into actionable 
tasks, so that whatever comes next (Claude Code, manual coding, Linear, 
Notion) works better." 
 
Positioning 
Archway bukan pengganti Excalidraw, Linear, atau Claude Code. Archway adalah layer yang duduk 
SEBELUM semua tools tersebut:

3. Problem Statement & Vision 
3.1 Problem Statement 
Developer, terutama solo developer dan freelancer, menghadapi tiga masalah yang saling 
terhubung: 
 
Problem 1: Skipping Pre-Coding (The Vibe Coding Trap) 
Developer langsung loncat ke coding tanpa dokumentasi. Hasilnya: edge case baru ketahuan di 
tengah jalan, database harus direfactor, scope creep, dan kehilangan arah. 
 
ARCHWAY  —  Pre-Coding Documentation v2.0

Problem 2: Fragmented & Boring Documentation Tools 
Tools yang ada (Excalidraw, Draw.io, Notion) terpisah-pisah, nggak ada guided flow, dan terasa 
seperti mengerjakan tugas kampus. Hasilnya: developer malas bikin dokumentasi. 
 
Problem 3: Documentation ≠ Action 
Bahkan kalau developer bikin dokumentasi, hasilnya sering jadi pajangan. Nggak ada jembatan 
otomatis dari dokumen ke task list. Developer harus manual translate PRD ke tasks (double work), 
dan sering task-nya nggak lengkap karena PRD-nya kurang detail. 
 
3.2 Vision 
"Satu platform visual yang memandu developer melewati seluruh tahap pre-
coding, menghasilkan dokumentasi lengkap DAN actionable task list yang siap 
dipakai di tool manapun — menjadikan proses pre-coding engaging, bukan 
membosankan."

Relationships 
• PROJECTS → NODES: 1:N (satu project punya banyak nodes) 
• PROJECTS → EDGES: 1:N (satu project punya banyak edges) 
• NODES → NODE_CONTENTS: 1:1 (setiap node punya tepat satu content) 
• NODES → TASKS: 1:N (satu node bisa generate banyak tasks) 
• PROJECTS → TASKS: 1:N (satu project punya banyak tasks, termasuk manual) 
 
Design Decisions 
• NODE_CONTENTS dipisah dari NODES: content bisa besar, pipeline rendering nggak perlu 
load semua content. 
• TASKS punya source_node_id: supaya user tahu task ini dari mana (traceability). Juga 
dibutuhkan saat regenerate. 
• is_manual flag: saat regenerate, hanya tasks dengan is_manual=false yang di-replace. 
Manual tasks aman.

14.4 Regeneration Logic 
When user clicks "Regenerate Tasks": 
• Confirm dialog: "This will re-generate auto-generated tasks. Your manually added tasks will 
be preserved." 
• Delete all tasks where is_manual = false AND project_id = current 
• Re-run all extraction rules on all nodes 
• Insert new generated tasks 
• Manual tasks (is_manual = true) untouched 
 
14.5 Future: AI Enhancement (Phase 3) 
Rule-based extraction handles 80% of cases. For the remaining 20% (context-aware tasks, 
dependency ordering, effort estimation), an LLM can analyze all docs holistically and generate 
smarter tasks. This is a future enhancement that layers on top of the rule-based engine.

Appendix A: Pain Point → Solution Detailed Mapping 
Detail lengkap setiap pain point, dampaknya, dan bagaimana Archway menyelesaikannya. 
 
P1: Scope Creep 
Problem: Fitur nambah terus karena nggak ada batasan jelas di awal. Developer bilang "ya" ke 
setiap request tanpa evaluasi dampaknya terhadap timeline dan arsitektur. 
Impact: Timeline molor, arsitektur jadi messy, technical debt menumpuk, developer burnout. 
Archway Solution: Project Brief node mendefinisikan scope di awal. Requirements node memaksa 
prioritisasi (MoSCoW). Kalau mau tambah fitur, harus kembali ke Requirements node dan conscious 
decision: apakah ini Must, Should, Could, atau Won’t? Ini menciptakan friction yang sehat terhadap 
scope creep. 
Related Features: FR-23, FR-24 
 
P2: Kehilangan Arah 
Problem: Di tengah project, developer lupa mau ngerjain apa, apa yang sudah selesai, apa yang 
belum. 
Impact: Kerja nggak fokus, bolak-balik antar fitur tanpa menyelesaikan satupun, burnout. 
Archway Solution: Visual pipeline dengan node status (Empty/In Progress/Done) = peta progress 
yang selalu terlihat. Task Board node = checklist yang bisa dicek kapan saja. Kalau bingung, lihat 
pipeline dan Task Board. 
Related Features: FR-15, FR-32 
 
P3: Database Refactoring 
Problem: Bikin tabel database asal-asalan tanpa ERD, lalu harus refactor berkali-kali saat ketemu 
kebutuhan baru. 
Impact: Data migration yang risky, bug karena schema berubah, waktu terbuang. 
Archway Solution: ERD node memaksa developer memikirkan entities, attributes, dan relationships 
SEBELUM menulis migration pertama. Flowchart dan Sequence Diagram juga membantu validasi 
apakah ERD sudah cover semua alur. 
Related Features: FR-26 
 
P4: Fragmented Tools 
Problem: ERD di dbdiagram.io, flowchart di Draw.io, notes di Notion, requirements di Google Docs. 
Semua terpisah. 
Impact: Konteks terpecah, sulit melihat gambaran besar, informasi outdated di beberapa tempat. 
Archway Solution: Satu workspace: semua docs dan diagrams di satu tempat, saling terhubung 
lewat pipeline. Update ERD? Task Board otomatis update juga. 
ARCHWAY  —  Pre-Coding Documentation v2.0 
Archway — Better Input, Better Output  |  Page 24 
Related Features: FR-10, FR-11 
 
P5: Nggak Tahu Mulai dari Mana 
Problem: Developer tahu harus bikin dokumentasi tapi nggak tahu urutan yang benar. Harus bikin 
apa dulu? ERD atau flowchart? 
Impact: Paralysis, akhirnya nggak bikin dokumentasi sama sekali dan langsung coding. 
Archway Solution: Guided pipeline dengan recommended order (Brief → Requirements → Stories 
→ Use Case → Flowchart → DFD → ERD → Sequence). Soft guidance, nggak di-lock, tapi 
developer tahu jalur yang ideal. 
Related Features: FR-11, FR-16 
 
P6: Dokumentasi Itu Boring 
Problem: Bikin dokumentasi terasa seperti tugas kampus, bukan "building something". Nggak ada 
dopamine hit. 
Impact: Developer malas dan skip dokumentasi, langsung vibe coding. 
Archway Solution: Node-based visual experience. Setiap node yang selesai = visual progress 
(badge berubah hijau). Pipeline yang terlihat makin "completed" memberikan sense of progress. 
Bukan blank document, tapi interactive workspace. 
Related Features: FR-10, FR-15, FR-22 
 
P7: Dokumentasi Nggak Actionable 
Problem: Bahkan kalau bikin dokumentasi, hasilnya cuma jadi pajangan. Nggak ada jembatan 
langsung ke coding tasks. 
Impact: Dokumentasi terasa sia-sia, reinforcing belief bahwa "docs itu buang waktu". 
Archway Solution: Auto task generation: setiap node yang diisi langsung menghasilkan tasks di 
Task Board. Dokumentasi bukan lagi pajangan, tapi langsung menjadi action items. 
Related Features: FR-30, FR-31, FR-32 
 
P8: Double Work (Docs → Tasks) 
Problem: Setelah bikin PRD, developer harus manual translate ke task list. Ini double work yang 
membosankan. 
Impact: Waktu terbuang, task list sering nggak lengkap karena malas translate semua. 
Archway Solution: Progressive task suggestion: saat mengisi User Stories, tasks langsung muncul. 
Saat mengisi ERD, database tasks langsung muncul. Zero double work. Export ke 
MD/CSV/JSON/Linear CSV langsung siap. 
Related Features: FR-30, FR-42, FR-43, FR-44, FR-45

P9: Susah Estimasi Effort 
Problem: Tanpa task breakdown yang granular, sulit mengestimasi berapa lama project akan 
selesai. 
Impact: Deadline missed, client kecewa, developer stress. 
Archway Solution: Task generation dari semua diagram types menghasilkan breakdown yang 
granular. Dari ERD muncul DB tasks, dari flowchart muncul implementation tasks, dari sequence 
muncul API tasks. Total tasks = rough effort estimate. 
Related Features: FR-31, FR-33 
 
P10: No Source of Truth 
Problem: Saat coding, nggak ada satu tempat yang bisa dirujuk untuk "apa requirement-nya" atau 
"gimana flow-nya". 
Impact: Developer bikin asumsi yang salah, fitur nggak sesuai requirement, rework. 
Archway Solution: Archway = single source of truth. Summary node compile semuanya. Exportable 
sebagai PDF (bisa diprint/buka di tab) atau Markdown (bisa commit ke repo). Selalu ada anchor. 
Related Features: FR-27, FR-40, FR-41
